## 1. 제품 개요

### 1.1 제품명(가칭)

- **툴매치(ToolMatch)** / **AI툴 나침반** / **VibeTool Finder** (배포 전 확정 예정)

### 1.2 한 줄 정의

비개발자·AI 초심자가 **몇 가지 질문에 답하면**, 상황에 맞는 **AI 도구를 추천**받고, **어떻게 시작하면 되는지**까지 한눈에 볼 수 있는 웹 서비스.

### 1.3 문제(Problem)

- AI 도구가 많아져 **무엇을 골라야 할지 판단 기준이 없음** (특히 비개발자·초심자)
- 숫자/점수 기반 추천은 **근거가 불명확해 신뢰가 떨어짐**
- "좋다는 건 알겠는데 어디서 어떻게 시작해야 하는지" **첫 발걸음이 막힘**

### 1.4 해결(Solution)

- **2~4분 서베이**로 사용자 상황 분류
- **점수 없이** 상황 기반 추천 + **근거(키워드)** + **사용법 가이드**
- 결과를 **공유 링크 / PNG**로 저장·공유 가능
- **웹 공개** 후 **피드백 수집**으로 개선 사이클

## 2. 타깃 사용자 & 유즈케이스

### 2.1 Primary 타깃


| 대상                 | 한 줄 니즈                           |
| ------------------ | -------------------------------- |
| 비개발자(창업자/PM/기획/전략) | AI 도구를 써보고 싶은데 뭐부터 시작해야 할지 모른다   |
| AI 초심자             | 이름은 들었는데 내 상황에 맞는 게 무엇인지 모른다     |
| 주니어·솔로 개발자         | 개발 속도를 올리고 싶은데 어떤 도구 조합이 맞는지 모른다 |


### 2.2 핵심 유즈케이스 (Top 3)


| ID  | 유즈케이스                               |
| --- | ----------------------------------- |
| U1  | "내 상황에 맞는 AI 도구가 뭔지" **3분 안에** 추천받기 |
| U2  | "추천받은 툴을 **어떻게 시작하면 되는지**" 바로 확인하기  |
| U3  | 결과를 **한 장 리포트**로 저장·공유하기 (팀 설득, 회고) |


### 2.3 공개·피드백 목표

- **웹에 공개**하여 누구나 접근 가능하게 운영
- **피드백 수집**: "도움 됐다"/"아쉽다" 간단 반응 + (선택) 자유 의견 → Google Form 연결
- 수집된 피드백은 추천 로직·문구·툴 카드 개선에 활용

## 3. MVP 범위

### 3.1 반드시 포함 (In scope)


| 영역          | 요구사항                                                                    |
| ----------- | ----------------------------------------------------------------------- |
| **서베이**     | 8~~12문항, 2~~4분 내 완료 가능                                                  |
| **결과 페이지**  | 추천 조합(Primary / Also), 툴별 키워드(자주 칭찬/불만), **툴 사용 시작 가이드**, 피칭 멘트(템플릿 기반) |
| **공유·내보내기** | 공유 링크(URL 파라미터 기반), PNG 다운로드                                            |
| **언어**      | UI 기본 **한국어** (영어는 후순위)                                                 |
| **피드백**     | 결과 페이지 하단 피드백 버튼 → Google Form 연결                                       |


### 3.2 제외 (Out of scope for MVP)

- 실시간 가격 비교·최저가 추적
- 커뮤니티 데이터 자동 크롤링 갱신 (초기에는 **수동 큐레이션**)
- 로그인·회원가입 (익명 결과 링크 중심)
- LLM API 호출 (추천·피칭 모두 룰/템플릿 기반으로 처리)
- 서버 저장소 (결과는 URL 파라미터로 유지)
- 2×2 포지셔닝 맵, 페르소나 매트릭스 (Post-MVP)

## 4. 서베이 설계

**원칙:** "정밀 진단"이 아니라 **초보자가 납득할 수 있는 분기**가 목표.

### 4.1 질문 구성 (8~12문항)


| #   | 카테고리    | 질문(요지)        | 선택지 예시                                                                   |
| --- | ------- | ------------- | ------------------------------------------------------------------------ |
| 1   | 역할      | 당신의 주 역할은?    | PM/기획/전략, 개발, 리드/매니저, 창업자, 기타                                            |
| 2   | 목표      | 가장 급한 목표는?    | A) 아이디어→데모/MVP, B) 기존 코드 개발 속도↑, C) 리서치/기획/문서 생산성↑, D) 병렬 에이전트로 작업 분산·운영 |
| 3   | 코드 편안함  | 코드 수정이 가능한가?  | 전혀 / 간단히 / 익숙 / 전문                                                       |
| 4   | 팀 구성    | 팀 규모는?        | 혼자 / 2~3명 / 팀(4명+)                                                       |
| 5   | 산출물     | 원하는 결과는?      | 데모 링크 / 깃허브 레포 / PRD·리포트 / 자동화된 PR                                       |
| 6   | 제약(보안)  | 보안·컴플라이언스 민감도 | 낮음 / 중간 / 높음                                                             |
| 7   | 제약(생태계) | 선호 생태계        | Google 중심 / OpenAI 중심 / 상관없음                                             |
| 8   | 시간      | 결과가 필요할 때     | 오늘 / 이번 주 / 이번 달                                                         |


(문항 수는 UX에 따라 8~12개로 조정 가능)

### 4.2 결과 분류 축(서베이 → 추천 입력)

- **Persona:** Idea→Demo / PM-Research / Solo Dev / Team Lead
- **Skill:** Code Comfort (Low / Med / High)
- **Constraint:** Security (Low / High), Ecosystem (Google / OpenAI / Any)
- **Output Type:** Demo / Repo / Doc / Automation

## 5. 추천 로직

### 5.1 원칙

- **점수화 없음** (근거 논쟁 방지)
- **룰 기반 매칭** (LLM API 호출 없음) — `rules.json` 정적 파일로 관리
- 결과는 **Primary 1~~2개 + Also 1~~2개** 조합
- 추천 이유는 **미리 작성된 템플릿 문장** (Persona + 조건 조합으로 자동 채워넣기)

### 5.2 예시 규칙 (초안)


| 조건                                     | Primary                   | Also                 |
| -------------------------------------- | ------------------------- | -------------------- |
| Persona = Idea→Demo, CodeComfort = Low | Lovable                   | ChatGPT / Claude     |
| Persona = Solo Dev, CodeComfort = High | Cursor 또는 Claude Code     | ChatGPT / Claude     |
| Persona = Team Lead                    | Codex 또는 Antigravity      | Cursor / Claude Code |
| Persona = PM-Research                  | ChatGPT / Claude / Gemini | Lovable(프로토타입)       |


(실제 구현 시 `rules.json`에서 확장·세분화)

## 6. 결과 페이지 정보 설계

### 6.1 블록 구성


| 순서  | 블록                  | 내용                                                  |
| --- | ------------------- | --------------------------------------------------- |
| 1   | **상단 요약 카드**        | "당신은 [Persona] 유형", "가장 잘 맞는 조합", "추천 이유 2~3줄(템플릿)" |
| 2   | **툴 카드 (키워드 클러스터)** | 툴별 "자주 칭찬" / "자주 불만" + "이 툴이 잘 맞는 상황" 1줄            |
| 3   | **툴 사용 시작 가이드**     | 추천된 툴별로 "이렇게 시작하세요" 3단계 안내 (정적 콘텐츠)                 |
| 4   | **피칭용 스크립트**        | 20초 버전, 복사 버튼 (템플릿 기반, LLM 미사용)                     |
| 5   | **공유·내보내기**         | 링크 복사(URL 파라미터), PNG 다운로드                           |
| 6   | **피드백**             | "도움 됐어요" / "아쉬워요" 버튼 → Google Form 연결               |


### 6.2 UI/UX 참고

- 시각 스타일: 기존 **ai_tools_keyword_clusters_en.html** 샘플과 동일한 톤 (다크 테마, 카드·칩 그리드)
- 한글 표기 및 가독성 우선

## 7. 콘텐츠·데이터

### 7.1 툴 데이터 구조 (`tools.json`)

- **name**, **category** (builder / IDE / agent / chat)
- **best_for** (문장 1줄 — "이 툴이 잘 맞는 상황")
- **praised_keywords[]**, **criticized_keywords[]**
- **persona_fit** (P: 주도 / A: 보조)
- **how_to_start[]** (3단계 시작 가이드, 각 단계 1~2줄)
- **references[]** (공식 문서·시작 튜토리얼 링크)

### 7.2 콘텐츠 작성 기준

- **톤:** 비개발자·초심자 기준, 전문 용어 최소화
- **분량:** 툴 카드 1개당 칭찬/불만 키워드 각 3~5개, 사용 가이드 3단계
- **업데이트 주기:** 월 1회 이상, 파일 상단에 `last_updated` 기록
- **출처:** 커뮤니티(Reddit, X, 유튜브 댓글) 기반 수동 큐레이션, 원문 링크 보존

### 7.3 운영 방식 (MVP)

- `**tools.json` + `rules.json` 정적 파일로 관리** → GitHub 버전 관리
- 결과 페이지에 **"마지막 업데이트 날짜"** 노출로 신뢰 강화
- MVP는 주요 툴 **5~8개**로 시작, 피드백 기반 확장

### 7.4 참고 자료 (내부 문서)

- **대표 AI 모델·서비스:** `docs/reference/ai-models-and-services.md` — 툴 목록·분류·한 줄·강점·한계·참고 링크.
- **리뷰 수집·추천 근거:** `docs/reference/review-sources-and-evidence.md` — 리뷰 소스, 키워드 클러스터 도출, 추천 문구·저작권 원칙.

## 9. 기술 요구사항

### 9.1 MVP 필수

- **반응형** (모바일/데스크탑)
- **결과 링크 유지:** URL 파라미터로 서베이 결과 인코딩 (서버 저장 불필요)
- **PNG 내보내기:** 클라이언트 측 canvas 처리
- **LLM API 호출 없음:** 추천·피칭 모두 룰/템플릿 기반 (외부 AI API 의존 없음)
- **i18n** 최소 한국어, 구조는 영어 확장 가능
- **접근성** 기본 (명확한 폰트·대비)

### 9.2 스택

- **Front:** 정적 HTML/JS 또는 Vite+React
- **데이터:** `tools.json` + `rules.json` 정적 파일 (GitHub 관리)
- **피드백:** Google Form (별도 저장소 불필요)
- **배포:** Vercel 또는 GitHub Pages (정적 호스팅)
- **Export:** 클라이언트 측 canvas → PNG
- **폰트:** Noto Sans KR 등 한글 웹폰트
